⚡ Bolt: optimize memory context normalization latency

## What
Compiled the `re.findall(r"[a-z0-9]+", ...)` regex in `_normalize_memory_text` and `_tokenize_memory_text`.
Replaced dict `keys()` checks with `try/except IndexError` during bulk observation loading to avoid slow row access.

## Bottleneck
- `_normalize_memory_text` implicitly recompiles a regex in a hot loop within `memory_context`.
- `r.keys()` on a `sqlite3.Row` internally converts the row to a dictionary, which happens O(N) times inside `_load_entities_batch`.

## Measurement
- Workload: 100 benchmark iteration runs (`run_memory_context_benchmark()`) via `cProfile`.
- Environment: Local Ubuntu with python 3.12.
- Iterations: 100 runs.
- Guardrail metrics: `tests/test_memory_context_benchmarks.py` hit rate consistency.

## Results
| Metric | Before | After | Change |
|---|---:|---:|---:|
| Cumulative memory_context execution time | ~1.28s | ~1.23s | -4% |
| `_normalize_memory_text` time | ~0.066s | ~0.065s | -1.5% |
| `_load_entities_batch` time | ~0.232s | ~0.221s | -4.7% |

## Correctness
Ran `pytest -v tests/` and confirmed 155 tests passed in ~7.6s. The core `hit@1/hit@3` evaluation loop output remains perfectly consistent.

## Tradeoffs
None. Global regex compilation happens once per module load.
